## Findings

**Setup.** 12 live runs against GPT-5.2 over NASA VEDA STAC (`lis-global-da-gpp`, Mendocino County CA, June–November 2020). 2 conversation patterns × 3 response-handler modes × 2 prompt variants (natural vs forced `compute_stats`). 11/12 runs completed; one (`eie × key-fields × forced`) hit the 30-turn recursion limit.

**1. The paper's ~7.5× templating lever is directionally real but overstated for the comparisons most production teams would actually run.**

The headline C/A ratio (raw STAC blob → terse status summary) varies sharply by conversation pattern:

- Paper pattern (parallel ReAct, 4–7 turns): **2.27–2.69×**
- EIE pattern (gated drill-down, 11–15 turns): **4.65–6.21×**

The 6.21× number (EIE × forced) is the closest match to the paper's anchor but still 17% below it. The paper's own scenario shape (parallel ReAct) produces a much weaker lever because the agent terminates faster — there are fewer tool round-trips for templating to amortise over.

**2. The realistic production lever (C vs B — raw STAC vs typed key-fields templating) is meaningfully smaller: 3.08–5.98×.**

Most production agents already template responses to some extent (extracting `id`, `title`, `bbox`, and a primary asset URL is standard). Comparing raw passthrough to that realistic baseline — not to a degenerate status string — the lever drops to a 3–6× range. A team currently doing key-fields templating cannot expect another 7.5× by going further; they can expect to land somewhere around mode A's $0.02–$0.07/query band, which is only marginally cheaper than mode B's $0.024–$0.044.

**3. Status-only mode has an information-bottleneck cost that pure token-counting misses.**

In the initial dry run, `pattern-paper-status-only` (natural) terminated at 3 turns *without ever calling* `search_items` or `compute_stats` — the agent literally couldn't extract collection ids from "search returned N items" summaries, so it produced a hand-wavy final answer instead. The bench captures this by adding a `--force-compute-stats` instruction: when forced, paper-pattern × status-only's cost almost doubles ($0.035 → $0.065) because the agent burns extra retries trying to reconstruct missing context. Token math alone makes mode A look cheaper than it actually is when you require task completion.

**4. Prompt cache hit rates (44–80%) dominate absolute cost.**

GPT-5.2's cached-input pricing is ~10% of fresh input. Across modes, 50–67% of all input tokens were cached. This compresses the absolute spend dramatically — every scenario lands in $0.02–$0.15/query — and means the *relative* lever between modes is meaningful while the *absolute* monthly figures should be treated as workload-specific.

**5. EIE × key-fields × forced was the only pathological run.**

The agent retried `compute_stats` repeatedly after raster-API errors and exhausted the 30-turn recursion limit. The recoverable-error path I added (tool exceptions surface to the LLM as message content) helps the agent self-correct in most cases, but in the gated EIE workflow with the forced-compute-stats instruction the agent kept oscillating between gate questions and stat retries. Production agents should bound retry depth per tool, not per conversation.

**Limitations.** Single dataset, single AOI, single model. The paper's worked example used a different dataset (MiCASA Land Carbon Flux v1) that isn't in VEDA's catalog; we substituted LIS Global GPP as a like-for-like carbon-flux measure but the exact numerics will shift with dataset semantics. Cache rates depend on whether the same agent fleet runs back-to-back; an isolated cold-start would push absolute costs higher.

**Recommendation.** Quote the lever as **~3–6× for realistic templating choices** (raw → key-fields) and **up to ~6× for aggressive templating** (raw → status-only) — keeping the original 7.5× anchor as the upper end of an extreme comparison, not the typical case. The information-bottleneck failure mode in status-only deserves a sentence; it's the kind of detail that makes a benchmark feel honest.
